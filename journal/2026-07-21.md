## Interview-coaching signals: what else is well-established that we're missing

## The brand/logo saga

Robin asked for wordmark concepts, "lean on font, elegant." What followed was a real lesson in getting small visual details exactly right rather than approximately right, across several rounds:
- The mark was floating above the wordmark's actual baseline — root cause: `align-items: center` centers by full box height, and "vocare" has no descenders, so the invisible descender space in the line-height was throwing off the optical center. Fixed by switching to `align-items: baseline`.
- Asked to fix rounded end-caps that looked clipped. First attempt changed the curve's underlying bezier control points — which, without meaning to, changed the whole silhouette from a peaked mountain shape to a smooth dome. Robin caught this immediately and was right to be frustrated (*"why did you change the design, i just wanted rounded ends"*) — reverted exactly to the prior version rather than continuing to theorize.
- Root cause turned out to be a genuinely small thing: a degenerate bezier control point (the first control point sitting exactly on the path's start point) gave the curve an ambiguous tangent that a round linecap couldn't resolve cleanly. Fixed properly this time by overlaying small filled circles at the endpoints instead of touching the curve — but the first attempt at *that* fix also (wrongly) touched the inner arc, which Robin had explicitly not asked for (*"did i tell you to touch the inner arc? no, i didn't"*) — reverted that part immediately.
- Final real bug, and the most satisfying catch: the filled circles were being **clipped by the SVG's own `viewBox`** — the circle's radius pushed it slightly past the viewBox's bottom edge, and SVG clips at that boundary by default. Fixed by giving the viewBox a little more room. Robin had guessed this exact cause before I'd confirmed it.
- Applied the exact same tuned values (not re-approximated ones) into the actual PDF cover page later, after first getting the scaling proportions wrong there too (mismatched aspect ratio, an independently-guessed gap instead of a scaled one) — fixed by scaling every dimension by the same factor from the known-good header values.

### Scaffolding

### A real build-order bug, caught before it shipped

`@vocare/shared`'s `package.json` points `main`/`types` at its compiled `dist/`, which doesn't exist right after a fresh `npm ci` — so `tsc --noEmit` on `backend`/`web` failed on a clean checkout. This is exactly what CI runs on every push, so it would have failed there first if it hadn't been caught here. Caught by deliberately wiping `node_modules` and `dist` everywhere and re-running the full pipeline locally before trusting it, not by assuming a green local run meant CI would be green too. Fixed two ways: a root `postinstall` script that builds `shared` right after every install, and (belt-and-suspenders, since Railway's own build phase timing wasn't something to just assume) making the Railway build command build `shared` explicitly rather than lean on the hook alone.

### OpenSpec: a real drift between the spec's wording and the installed tool

The spec's M0 task says "OpenSpec init, anchor document (`project.md`) capturing Section 1's thesis" — written before anyone had checked what the *current* OpenSpec CLI actually does with project context. Running `openspec init --tools claude` (CLI 1.6.0, the real package, confirmed again) produced `openspec/config.yaml` with a commented-out `context` field and no mention of `project.md` anywhere in the generated `.claude/skills/` files. Grepped for "project.md" across everything OpenSpec generated — zero hits. The current mechanism is `config.yaml`'s `context:` field, read by `openspec instructions` when an artifact is created. Wrote Section 1's product thesis there instead of creating a file nothing would actually read — a smaller, more accurate echo of Section 9's own earlier finding that the spec's assumed OpenSpec CLI syntax (`openspec change create`) didn't match the real tool either.

### CI, verified against a clean checkout before trusting it

### Deploying to both platforms — and a real lesson repeated twice

First attempt at both Railway and Vercel used the "deploy from the subdirectory" shortcut (`railway up backend --path-as-root`-style thinking for Railway; `vercel deploy web` for Vercel) — the intuitive move for "I only want to deploy this one app." Checked Railway's actual monorepo docs before running it (not from memory) and found the documented answer directly: for services with sibling-workspace dependencies, Railway calls this a "shared monorepo" and the fix is custom build/start commands run from repo root, not a "Root Directory" setting (that's only for fully isolated monorepo services with no cross-directory dependencies). Applied that first, correctly, for Railway: root `railway.json` plus `build:backend`/`start:backend` scripts, deployed from repo root, verified live at `/health`.

Vercel got the lesson the harder way — ran `vercel deploy web --yes --prod` without checking first, and it failed for exactly the same underlying reason Railway's docs had just explained: Vercel's CLI, given a subdirectory, only uploads that subdirectory (188KB, just `web/`), so `npm install` couldn't see `@vocare/shared` at all and failed outright. Real, verified: deployed only after removing the misconfigured project and redeploying from repo root with a root `vercel.json` (`outputDirectory: web/dist`) — the exact same pattern that had already worked for Railway, which in hindsight should have been applied the first time instead of re-learning it. Vercel also auto-connected the GitHub repo during this deploy, so pushes to `main` now trigger automatic deployments — a real, standing side effect worth remembering, not something explicitly asked for.

One more small thing caught in passing: `package.json`'s `engines.node` was pinned to the exact `24.18.0` patch, which threw a (non-fatal) `EBADENGINE` warning on Vercel's build image (running `24.15.0`, a different patch on the same major LTS line). Loosened to `>=24.0.0 <25.0.0` — still enforces the right major version, stops the noise on every future deploy.

Both are live and verified by actually curling them, not by trusting the deploy tool's "success" message alone: backend's `/health` returns `{"status":"ok","app":"Vocare"}` on Railway; the web app's title tag reads "Vocare" (fixed from Vite's default "web" title) on Vercel.

## Session 3 — vocare.ca goes live

Back in the original chat session (not a fresh one — this was a short, tightly-scoped continuation to wire up a domain that had just been purchased, not new/unrelated work). `vocare.ca` was purchased the morning of this same day; DNS had never been touched.

**Registrar confirmed: Web Hosting Canada (WHC.ca).** The domain was still sitting on WHC's default parking nameservers (`parking1.whc.ca`/`parking2.whc.ca`) and default parking A record — meaning DNS itself was already hosted at WHC, so no nameserver change was needed, just an edit inside WHC's own DNS Zone Editor.

**Vercel side:** added `vocare.ca` under the existing `vocare` project's Domains settings (Add Existing, not Buy — already owned). Vercel's default offered to redirect the apex to `www` — declined that and kept the apex canonical instead, matching the clean/minimal brand direction from earlier in the day (Stripe/Linear/Notion-style, no `www.` needed for a modern brand). Vercel then displayed its actual current target IP for the A record: `216.198.79.1` — deliberately not asserted from memory, since Vercel's specific anycast IPs are the kind of thing that changes over time and its own dashboard is the only reliable source at the moment of setup.

**WHC side:** the actual DNS record editor wasn't obviously placed — took three tries to find (checked "Nameservers" first, wrong section entirely; almost changed nameservers by mistake, caught and cancelled before applying; found it via the gear icon next to "Renew," landing on `.../vocare.ca/zone`). Once there: edited the *existing* default A record for `@` (WHC's own parking IP, `149.56.225.6`) to Vercel's `216.198.79.1`, rather than adding a new record. Left everything else in the zone untouched — the `www` CNAME already pointed at the apex (already matching the canonical-apex decision, no change needed), and the MX/TXT/`emailfwd` records are WHC's own default email forwarding, unrelated to this.

**A real, verified diagnosis, not a guess:** Vercel then reported `Failed To Generate Cert — the required http-01 challenge failed`. Rather than assume this meant something was misconfigured, checked actual DNS propagation directly (`nslookup vocare.ca` against Google's `8.8.8.8` and Cloudflare's `1.1.1.1`) and found exactly what would explain it: the local/default resolver already saw the new `216.198.79.1`, but both Google and Cloudflare were still serving the old, cached `149.56.225.6` — confirming genuine propagation lag (the old record's TTL was ~4 hours), not a real error. Waited, the two public resolvers caught up, Vercel's cert issuance succeeded on retry, and `vocare.ca` now serves the live M0 hello-world directly — confirmed by actually loading it in a browser, not just trusting Vercel's dashboard status.

**Deliberately not done yet, and not urgent:** `api.vocare.ca` (Railway's custom domain for the backend) and a dedicated subdomain for Resend's sending domain (e.g. `mail.vocare.ca`). Neither blocks anything today — Railway's default `*.up.railway.app` URL works fine functionally, and Resend's DNS records aren't needed until M1 actually needs to send a real magic-link email. Worth doing both before M1's auth work needs working email, given DNS/cert propagation is exactly the kind of thing that just cost real time today and shouldn't be a surprise blocker later.
