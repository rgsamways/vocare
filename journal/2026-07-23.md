### OpenSpec proposal

Full four-artifact proposal (`proposal.md`, `design.md`, `specs/coaching-feedback/spec.md`, `tasks.md`) written against `openspec/changes/m5-coaching-feedback/`, following M4's own archived artifacts as the template for this repo's conventions. Key decisions made and recorded in `design.md`:

- **Coaching notes are a deterministic template lookup, not a second LLM call.** `session_mining_results`' fields are already structured booleans/enums/arrays — turning them into plain-language notes is a lookup, not a generation task. Cheaper, and structurally can't drift into judgmental or scored phrasing under a bad prompt (the risk a second Haiku call would reopen).
- **v1's note catalog is narrower than the full mining schema on purpose.** Ownership language, tradeoff reasoning, outcome mention, clarity, quantified-impact quotes, and audience-keyword quotes are covered; `sentiment` and `techDomainMentions[]` are deliberately held out — `sentiment` in particular risks reading as the app commenting on a user's emotional state rather than their described work, closer to unsolicited pop-psychology than coaching. Flagged as an open question, not silently dropped.
- **`topicRelevanceScore` is excluded from the note-generator's input type itself**, not just left out of the rendering — a leak would require changing the function's signature first, not a rendering mistake.
- **No-match cases render as a normal, complete report.** Null `audienceKeywordMatches` or empty quote arrays simply omit that note category — never a "we didn't find anything" note, which would be the same disguised-negative-signal problem the spec's trend-reporting section already named, just arriving one module early.
- **Generation is chained onto M4's existing async mining seam**, not a new trigger — `mineSession` writes `feedback_reports` right after its existing `session_mining_results` insert, same try/catch, no new infrastructure.
- **Storage uses the `sessionId`-as-primary-key convention** M4 already established for `session_mining_results`, for the same one-row-per-session reason, rather than the spec's literal `id (pk), session_id (fk)` shape.

### A real M3 regression, found mid-session, diagnosed by asking rather than guessing

While testing M5, Robin reported voice capture "doubling up words" — a genuine change in behavior, not something noticed since M3 shipped. First-pass description ("hanging on a word too long") pointed at one mechanism; Robin corrected it a message later to the opposite ("speaking too fast"), which matters because the two point at different bugs. Confirmed via `git log` that `ConversationPage.tsx`'s mic code hasn't been touched since M3 itself — ruling out a regression introduced by this project's own code and pointing at Chrome's own recognition service having shifted since M3 shipped (M3's own design doc already flagged this exact endpointer behavior as undocumented and liable to change).

Root cause theory: fast speech leaves less silence for Chrome's endpointer to cleanly delimit segment boundaries, and it appears to sometimes finalize the same segment twice as separate adjacent entries in `event.results`. Fix: collapse an exact, immediately-adjacent repeat of a whole result segment before assembling the composer text — targets the reported symptom without needing to fully reverse-engineer an undocumented browser quirk, at the small, explicitly-noted cost of also collapsing a genuine back-to-back verbatim repeat if a user ever said the exact same segment twice on purpose. New regression test added using the existing `FakeSpeechRecognition` test pattern, driving a fabricated duplicate-segment event and asserting the composer text comes out deduplicated.

**Not yet deployed.** This fix (like the rest of today's frontend work) lives only in the working tree — Vercel auto-deploys on push to `main`, and nothing has been committed yet today. Flagged directly to Robin rather than assumed either way.
