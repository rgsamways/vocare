### A real bug, caught immediately: port collision

`docker compose up` for the local Postgres (`postgres:16-alpine`, following CLAUDE.md's documented port `5433`) came up fine, but `drizzle-kit migrate` immediately started failing password auth against something — despite the exact right user/password. Direct `psql` from inside the container worked fine; from the Windows host, `psql -h 127.0.0.1 -p 5433` failed the *same* auth error even with the CLI's own native tooling, not just the app. That ruled out an app-level bug and pointed at the port itself. Checked Windows services directly: a native **PostgreSQL 17** service (not just the already-accounted-for 18) was already bound to `5433` — CLAUDE.md's original port choice had only checked for collisions with 18 and Monkeyback's container, both on 5432, and missed that 17 existed at all. Moved Vocare's container to `5434`, confirmed free, migrated cleanly, and corrected CLAUDE.md's Project Facts on the spot rather than leaving the stale port number sitting there as a landmine for a future session.

### The build itself

Built in `tasks.md`'s own section order — schema, magic-link auth, Stripe billing, session entitlement, account management, verification — checking each task off only once actually built *and* tested, not just written:

- **Schema (Drizzle + `drizzle-kit`).** `users` isn't a separate table — it's Better Auth's own generated `user` table extended with `entitlementStatus`/`dateOfBirth`/`country`/`paidAt` via `additionalFields`, generated through `@better-auth/cli generate` rather than guessed. Confirmed the real table names design.md flagged as needing verification: `user`, `session`, `account`, `verification` — matched the design doc's own guess, but now *verified*, which is the point. Added `stripe_payments` (payment_intent → userId) and `pending_signups` (email → dateOfBirth/country, bridging the gap between requesting a magic link and Better Auth actually creating the account on verify) — neither named in the original spec, both discovered as genuinely necessary during implementation, both called out explicitly in the relevant task's completion note rather than slipped in quietly.
- **Magic-link auth.** Age gate enforced *before* any email even sends, via a custom `/auth/pending-signup` pre-check — deliberately stronger than the spec's literal requirement ("blocks sign-up, doesn't create an account") since it also avoids issuing tokens to people who fail the age check at all. Session sliding window confirmed via real math on a real DB row (`expiresAt` sat within 29.9–30.1 days of verification), not just asserted from config.
- **Stripe billing.** Webhook idempotency implemented exactly as design.md prescribed — insert `event.id` first inside the transaction, let a unique-violation abort it, treat that specific Postgres error code (`23505`) as "already processed," rethrow anything else. Tested signature verification with Stripe SDK's own `generateTestHeaderString` — genuine HMAC round-trips, no live Stripe account needed for that part.
- **Session entitlement.** Velocity cap and free-session count both implemented as `COUNT(*)` queries with no mutable counter column, per design.md's stated reasoning (a counter can drift from the real session log; a computed count can't). The Android "call path" for M10 turned out to need zero platform-specific code — it's the same `GET /entitlement/check` REST endpoint web will also call, which is the whole point of "identically enforced on both platforms."
- **Account management.** `deleteUserCascade` deletes across every table that exists as of M1; the six tables named in the spec that belong to *later* modules (`transcript_turns`, `session_mining_results`, etc.) don't exist yet, so the function has an explicit comment marking itself as the extension point — the same failure mode (a table added later, deletion list never updated) that already happened once with `anchors`, called out on purpose this time. `verification` rows aren't keyed by userId at all (Better Auth keys them by the token, storing email inside a JSON `value` column) — matched by an email `LIKE` pattern instead, otherwise a stale magic-link token for a deleted identity would survive deletion.

### Bugs found only by actually running it, not by reading the code

Two real bugs surfaced exclusively because of insisting on driving the app in a real browser (via a temporarily `npm install --no-save`'d Playwright, never persisted to `package.json`) instead of stopping at green tests:

1. **CORS silently broken on the hijacked `/api/auth/*` route.** `@fastify/cors` sets headers via `reply.header()`, flushed by Fastify's normal send pipeline — but the auth route calls `reply.hijack()` to hand the raw request/response to Better Auth's own Node handler, which bypasses that pipeline entirely. Every cross-origin request from the web app was failing with a CORS error that never showed up in any backend log, only in the browser console. Fixed by writing the CORS headers straight onto `reply.raw` before hijacking, plus handling the OPTIONS preflight inline. A second, narrower version of the same class of bug: `@fastify/cors`'s default allowed-methods list doesn't include `DELETE`, silently blocking the account-deletion button until made explicit.
2. **A real double-verify race, found via Robin's own inbox, not staged.** Robin received and clicked the actual Resend-delivered magic-link email — landed authenticated with correct data, but with a stray `?error=INVALID_TOKEN` sitting in the URL. Backend logs showed the *same* single-use token verified twice, three minutes apart, both from `127.0.0.1` — meaning it couldn't have been Gmail's server-side link scanner (no network path from Google's infrastructure to a `localhost` dev server), more likely Chrome's own local link-prefetch on the same machine as the dev server. Better Auth's single-use enforcement was working exactly as designed; the actual bug was the UI leaving a stale error visible despite a valid session already existing. Fixed with an effect that strips the query string once a session is confirmed, then re-verified by deliberately reproducing the exact double-verify sequence in a real browser rather than trusting the fix by inspection.

## `chat`'s independent grading, and a CI gap that made it to `main`

Read `proposal.md`/`design.md`/`tasks.md`/the four `specs/*.md` files cold, then verified rather than accepted: independently re-ran all three workspaces' test suites (21 + 1 + 1 = 23, matching the claim exactly) and `openspec validate --strict` myself, then read the actual implementation of both claimed bug fixes and the full Stripe webhook money-flipping path end to end — including confirming `checkout.ts` actually sets `client_reference_id`, the one thing that would have made the webhook's entitlement flip a silent no-op for every real user if it were missing. The Better Auth cascade-delete gap flagged in the prior grading pass was independently re-confirmed fixed, with a genuinely sharp detail (`verification` rows matched by email, not `userId`, since Better Auth keys them by token) that wasn't glossed over. Approved; archived via `openspec archive`, committed (74 files), pushed.

**Then CI failed on `main` — a real gap, not flakiness, and worth being honest about exactly how it slipped through.** `.github/workflows/ci.yml` was never updated to run a Postgres service; M1's tests (the first ones in this repo to touch a real database) connected fine locally against the already-running Docker container but had nothing to connect to in GitHub's runner, failing with `ECONNREFUSED` on port 5432. This is the same class of bug M0's own journal entry already named once — "would have failed in CI first if it hadn't been caught here" — except this time it wasn't caught first. Grading verified the code; it didn't verify the pipeline the code would actually ship through, and "I ran the tests locally" quietly meant "against my own already-migrated database," not "against what CI actually has."

Fixed properly, not just patched and hoped: added a `postgres:16-alpine` service block to the workflow (matching the local dev image) and a migration step — but the first version of that fix was *also* wrong, caught before pushing by actually testing it rather than trusting the YAML looked right. Spun up a genuinely fresh, empty Postgres container on a spare port to simulate CI's clean state, and `drizzle-kit migrate --config=backend/drizzle.config.ts` invoked from repo root failed silently (hung on its own spinner with no clear error) — while the exact same migration invoked from inside `backend/` with the default config applied cleanly. The working-directory difference, not the flag syntax, was the actual cause. Re-ran the full backend test suite against that freshly-migrated database (21/21 passing) before committing the real fix.

**The honest lesson, worth stating plainly rather than smoothing over:** verifying code and verifying the pipeline it ships through are two different checks, and skipping the second one is exactly how a fully-graded, fully-tested change still broke `main`.
