### Bugs found only by actually running it, not by reading the code

Two real bugs surfaced exclusively because of insisting on driving the app in a real browser (via a temporarily `npm install --no-save`'d Playwright, never persisted to `package.json`) instead of stopping at green tests:

1. **CORS silently broken on the hijacked `/api/auth/*` route.** `@fastify/cors` sets headers via `reply.header()`, flushed by Fastify's normal send pipeline — but the auth route calls `reply.hijack()` to hand the raw request/response to Better Auth's own Node handler, which bypasses that pipeline entirely. Every cross-origin request from the web app was failing with a CORS error that never showed up in any backend log, only in the browser console. Fixed by writing the CORS headers straight onto `reply.raw` before hijacking, plus handling the OPTIONS preflight inline. A second, narrower version of the same class of bug: `@fastify/cors`'s default allowed-methods list doesn't include `DELETE`, silently blocking the account-deletion button until made explicit.

Better Auth's single-use enforcement was working exactly as designed; the actual bug was the UI leaving a stale error visible despite a valid session already existing. ## `api.vocare.ca`'s TLS certificate, and a real mistake made and caught mid-fix

Robin reported Railway's custom domain failing to issue a TLS certificate — a generic "internal error, please retry" with the domain settings showing "Port 8080." Investigated rather than just clicking retry.

**A real mistake, caught and reversed immediately:** `railway status` showed an ambiguous linked-service context (Postgres, not the app), and running `railway domain` with no explicit `--service` flag to "just check the domain" instead **created a brand-new public domain for the Postgres database itself** — a real, unintended security exposure, not a cosmetic error. Caught by actually reading the command's output rather than assuming it did what was intended; deleted immediately (`railway domain delete ... --service Postgres`), confirmed gone via `railway domain list`.

**Then a second, more subtle mistake, also caught before it caused real harm.** `railway variables --service vocare` showed no explicit `PORT` variable, and the app's own code (`process.env.PORT ?? 3000`) implied it would default to 3000 — a real, concrete-looking mismatch against the domain's configured target port of 8080. Changed the domain's target port to 3000 to "fix" it. This was wrong: `railway logs` showed the actual running container listening on `8080`, meaning Railway injects `PORT` at the container runtime level in a way that never appears in the `variables` list (which only shows explicitly-set values, not platform-injected ones) — the original 8080 target was correct all along, and the "fix" introduced a real mismatch where none had existed. Reverted immediately back to 8080. The certificate issued successfully right after — possibly the revert itself, possibly just Railway's own retry mechanism catching up; genuinely unclear which, and worth being honest about that rather than claiming certainty either way.

**Worth stating plainly, the same way the CI gap above was:** two real, live-infrastructure mistakes happened in the course of investigating a single reported problem — one a scope/context error (wrong service), one a verification error (trusted `variables` output over `logs` output as the source of truth for a runtime-injected value). Both were caught by actually checking the result of each action before moving to the next one, not by getting it right on the first attempt.
