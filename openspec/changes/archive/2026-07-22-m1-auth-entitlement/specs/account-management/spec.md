## ADDED Requirements

### Requirement: Profile/account screen
The system SHALL provide a profile screen showing the user's email, current entitlement status in plain language, and an account-deletion trigger.

#### Scenario: Unpaid user sees remaining free sessions
- **WHEN** an unpaid user with free sessions remaining views their profile screen
- **THEN** the screen displays how many free sessions they have left

#### Scenario: Paid user sees unlock date
- **WHEN** a paid user views their profile screen
- **THEN** the screen displays that their account is unlocked and the date they paid

### Requirement: Full account-deletion cascade
Deleting an account SHALL remove that user's data from every table that holds it: `users`, `sessions`, `transcript_turns`, `session_mining_results`, `feedback_reports`, `tier1_profiles`, `anchors`, `anchor_revisions`, and Better Auth's own session/verification-token tables (the exact table names generated by its Drizzle adapter — confirmed against the real generated schema, not assumed). No table holding user-linked data SHALL be left with orphaned rows after deletion.

#### Scenario: Account deletion removes data across all tables
- **WHEN** a user triggers account deletion and confirms
- **THEN** all rows linked to that user across `users`, `sessions`, `transcript_turns`, `session_mining_results`, `feedback_reports`, `tier1_profiles`, `anchors`, `anchor_revisions`, and Better Auth's session/verification-token tables are removed

#### Scenario: Deleted account cannot sign in again under the same identity
- **WHEN** a previously deleted user attempts to use an old magic link or session token
- **THEN** the system rejects the attempt, since no account, session, or verification-token record for that user remains in either the app's own tables or Better Auth's tables
